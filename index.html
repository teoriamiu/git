Clase teórica de Git

Git no es simplemente una herramienta para “guardar versiones”. Es un sistema formal de control de versiones distribuido, diseñado para modelar la evolución del conocimiento técnico en entornos colaborativos bajo condiciones de conflicto, divergencia y cambio continuo.

Su marco teórico combina teoría de grafos, sistemas distribuidos y principios de inmutabilidad.

1. Ontología: qué “existe” en Git

La entidad fundamental en Git no es el archivo, ni la carpeta, ni el repositorio, sino el commit.

  • Un commit es un objeto inmutable.
  • Cada commit referencia a uno o más commits padres.
  • El historial forma un grafo acíclico dirigido (DAG).

Las ramas no son entidades físicas: son punteros móviles dentro del grafo.

2. El contenido antes que el archivo

Git no versiona archivos: versiona contenido.

  • Los archivos se representan como blobs.
  • Los directorios son árboles que referencian blobs y otros árboles.
  • El nombre del archivo es secundario frente al hash del contenido.

La identidad se define por lo que es, no por dónde está.

3. Inmutabilidad como principio fundacional

  • Un commit nunca se modifica.
  • Si el contenido cambia, el hash cambia.
  • El pasado no se edita: se crean nuevos nodos.

Conceptualmente, Git es un sistema append-only.

4. Versionado como grafo, no como línea

  • El historial puede bifurcarse.
  • Múltiples líneas de desarrollo coexisten.
  • La convergencia ocurre mediante merge o rebase.

El conflicto no es un error: es un estado normal del sistema.

5. Distribución como axioma

  • Cada clon es un repositorio completo.
  • No existe un servidor central ontológicamente privilegiado.
  • La autoridad es social, no técnica.

6. Epistemología del versionado

  • No existe un estado verdadero único.
  • Existen múltiples historias posibles.
  • La verdad emerge por consenso.

El control de versiones se convierte en una negociación explícita de significado.

7. Operaciones como transformaciones del grafo

  • Commit crea nodos.
  • Checkout mueve punteros.
  • Merge conecta grafos.
  • Rebase reescribe trayectorias.

8. Tensiones y límites

  • Modelo mental complejo.
  • Metáforas engañosas basadas en archivos.
  • Riesgos de reescritura histórica mal comprendidos.
  • Comandos potentes con efectos profundos.

9. Síntesis conceptual

“Un sistema distribuido de versionado basado en objetos inmutables organizados en un grafo, diseñado para gestionar la evolución divergente y convergente del contenido bajo condiciones de colaboración y conflicto.”

Git no intenta evitar el conflicto. Intenta hacerlo explícito, rastreable y reversible.
